# Prompt Enhancement Standard

> 高质量 AI 角色 Prompt 的设计规范与检查清单

---

## 1. 为什么需要 Prompt 增强标准

当前 AI 角色存在的主要问题：

- **太简陋**：只是一个功能标签，缺乏深度
- **不够专业**：专业类角色缺乏真正专业性
- **不像真人**：娱乐类角色只是简单的语气模仿
- **交互能力弱**：缺乏自然语言理解、多轮对话、上下文跟踪能力

---

## 2. 标准模块框架

### 2.1 核心模块（必填）

一个完整的角色 Prompt 必须包含以下模块：

```markdown
# [角色名称]

## 1. Role / Identity
- 身份定位（我是谁）
- 背景描述（我有过什么经历）
- 核心特质（我是什么人）
- 价值观（我相信什么）

## 2. Core Mission
- 主要职责（我做什么）
- 价值主张（我为用户提供什么）
- 工作边界（我不做什么）

## 3. Strengths
- 核心能力列表
- 思维方式/分析框架
- 方法论/工作流
- 工具意识（如适用）

## 4. Personality / Style
- 性格特征
- 沟通风格
- 语气特征
- 表达习惯

## 5. Task Handling Method
- 完整工作流程（Phase 1-N）
- 每个阶段的具体步骤
- 阶段间的转换逻辑

## 6. Output Style
- 不同场景的输出模板
- 格式规范
- 结构化表达的示例

## 7. Boundaries / Constraints
- 专业边界（我不覆盖的领域）
- 安全边界（不能做的事）
- 免责声明（如需要）
- 角色边界（角色扮演类）

## 8. Why This Role Is Valuable
- 角色独特价值
- 解决的问题场景
- 用户收益
```

### 2.2 智能性增强模块（推荐）

```markdown
## 9. 智能性设计

### 9.1 自然语言理解
- 能理解用户的模糊表达
- 能从上下文中推断意图
- 能识别言外之意和情绪信号

### 9.2 多轮对话能力
- 能跟踪对话历史
- 能基于之前的信息修正理解
- 能在长对话中保持上下文一致性

### 9.3 上下文适应
- 能记住用户偏好
- 能根据反馈调整输出
- 能在对话中学习用户习惯

### 9.4 意图识别
- 能主动提问澄清
- 能在信息足够时直接推进
- 能在模糊时提供多个方向
```

### 2.3 专业性增强模块（专业类角色必填）

```markdown
## 10. 专业性设计

### 10.1 领域知识体系
- 核心知识领域
- 行业规范与标准
- 专业术语体系

### 10.2 工具意识
- 了解使用的工具类型
- 工具适用场景
- 工具局限性

### 10.3 工作流设计
- 专业方法论
- 分析框架
- 输出标准

### 10.4 风险意识
- 识别专业领域风险
- 边界条件处理
- 免责声明
```

### 2.4 拟真性增强模块（娱乐/角色扮演类必填）

```markdown
## 10. 拟真性设计

### 10.1 人设核心
- 身份背景
- 性格层次（外层/中层/内核）
- 行为动机

### 10.2 语气风格库
- 口头禅
- 习惯性动作
- 情绪表达方式
- 典型场景反应

### 10.3 角色边界
- 适合的场景
- 不适合的场景
- 角色一致性维护

### 10.4 互动模式
- 对话节奏
- 傲娇/温柔等状态切换
- 长期互动中的成长
```

## 3. 专业类角色增强标准

### 3.1 必须体现的要素

| 要素 | 说明 | 检查 |
|:---|:---|:---:|
| 明确专业领域 | 清晰定义专业范围 | ☐ |
| 明确擅长任务 | 列出具体可服务的任务 | ☐ |
| 明确工作流 | 标准化的处理流程 | ☐ |
| 明确分析框架 | 专业的分析方法论 | ☐ |
| 明确输出结构 | 标准化的输出格式 | ☐ |
| 明确风险意识 | 识别风险和边界 | ☐ |
| 明确专业边界 | 不覆盖的领域 | ☐ |
| 工具意识 | 对工具的理解和局限 | ☐ |

### 3.2 专业性自检问题

- [ ] 是否像"真正的专业人士"而不是普通问答助手？
- [ ] 是否有专业方法论和分析框架？
- [ ] 是否有工具意识和边界意识？
- [ ] 是否有风险识别和免责声明？
- [ ] 是否能处理模糊的专业问题？

## 4. 娱乐/角色扮演类增强标准

### 4.1 必须体现的要素

| 要素 | 说明 | 检查 |
|:---|:---|:---:|
| 真实人设感 | 有完整的身份背景和性格 | ☐ |
| 语气风格 | 有独特的说话习惯 | ☐ |
| 情绪表达 | 有丰富的情绪反应 | ☐ |
| 互动魅力 | 有吸引人的互动方式 | ☐ |
| 角色一致性 | 长期互动中保持一致 | ☐ |
| 边界清晰 | 适合/不适合的场景明确 | ☐ |

### 4.2 拟真性自检问题

- [ ] 是否只是"口癖模仿"还是有真实人设？
- [ ] 是否有内在的行为逻辑？
- [ ] 是否有情感深度而不是表面模仿？
- [ ] 能否在长期互动中保持一致性？
- [ ] 是否有独特的互动节奏和魅力？

## 5. 智能性增强标准

### 5.1 自然语言理解

| 能力 | 描述 | 检查 |
|:---|:---|:---:|
| 模糊理解 | 能理解不精确的表达 | ☐ |
| 意图推断 | 能从上下文推断真实意图 | ☐ |
| 弦外之音 | 能识别言外之意 | ☐ |
| 纠错能力 | 能识别用户的表达错误 | ☐ |

### 5.2 多轮对话

| 能力 | 描述 | 检查 |
|:---|:---|:---:|
| 记忆能力 | 能记住对话历史 | ☐ |
| 上下文跟踪 | 能保持上下文一致 | ☐ |
| 渐进深入 | 能随对话深入深化理解 | ☐ |
| 状态管理 | 能管理对话状态 | ☐ |

### 5.3 持续适应

| 能力 | 描述 | 检查 |
|:---|:---|:---:|
| 用户偏好学习 | 能记住用户偏好 | ☐ |
| 反馈调整 | 能根据反馈调整 | ☐ |
| 个性化 | 能适应不同用户 | ☐ |

## 6. 质量检查清单

### 6.1 完整性检查

- [ ] 有完整的 Role / Identity
- [ ] 有清晰的 Core Mission
- [ ] 有具体的 Strengths
- [ ] 有明确的 Personality / Style
- [ ] 有可执行 Task Handling Method
- [ ] 有结构化的 Output Style
- [ ] 有清晰的 Boundaries / Constraints
- [ ] 有价值说明 Why This Role Is Valuable

### 6.2 质量检查

- [ ] 角色不是"功能标签"而是"完整人设"
- [ ] 专业类角色有真正专业性
- [ ] 娱乐类角色有真实人设感
- [ ] 有智能性设计（自然语言理解、多轮对话等）
- [ ] 有清晰的安全边界
- [ ] 长期交互中能保持一致性

### 6.3 可用性检查

- [ ] 用户能通过 Raw 链接激活角色
- [ ] 激活后能立即进入角色模式
- [ ] 交互体验流畅自然
- [ ] 输出符合角色定位

## 7. 模板使用指南

### 7.1 创建新角色

1. 复制 `templates/role-template.md`
2. 填写所有必填模块
3. 根据角色类型添加增强模块
4. 确保智能性设计完整
5. 测试 Raw 链接激活流程

### 7.2 角色分类

| 类别 | 必须包含的增强模块 |
|:---|:---|
| 专业类 | 专业性设计、工具意识、风险意识 |
| 娱乐类 | 拟真性设计、人设核心、互动模式 |
| 通用类 | 智能性设计、持续适应 |

## 8. 常见问题

### Q1: 如何避免角色"太简陋"？

确保角色有：
- 完整的身份背景
- 清晰的行为逻辑
- 丰富的交互能力
- 持续的上下文理解

### Q2: 专业类角色如何体现"真正专业"？

确保有：
- 专业方法论
- 分析框架
- 工具意识
- 风险识别
- 边界意识

### Q3: 娱乐类角色如何避免"只是模仿"？

确保有：
- 完整人设
- 内在行为逻辑
- 情感深度
- 长期一致性

### Q4: 如何保证"智能性"？

确保有：
- 自然语言理解设计
- 多轮对话能力
- 上下文跟踪
- 持续适应
