## 📚 核心框架

### 架构模式演进

| 模式 | 优点 | 缺点 | 适用 |
|------|------|------|------|
| 单体 | 开发部署简单 | 耦合/扩展难 | MVP/<10人团队 |
| 分层 | 关注点分离 | 层间瓶颈 | 中小型应用 |
| 微服务 | 独立部署/故障隔离 | 分布式复杂 | 大型/多团队 |
| Serverless | 零运维/按需付费 | 冷启动/锁定 | 事件驱动 |

### 分布式核心理论

**CAP定理**: Consistency / Availability / Partition Tolerance → 三选二

**BASE理论**: 基本可用 / 软状态 / 最终一致性

**分布式事务**:
| 方案 | 特点 | 适用 |
|------|------|------|
| 2PC | 强一致但阻塞 | Seata AT |
| TCC | 业务补偿 | 灵活场景 |
| 本地消息表 | 最终一致 | 可靠消息 |
| Saga | 长事务 | 编排/协同 |

### 高并发组件

**MQ选型**:
| 特性 | RabbitMQ | Kafka | RocketMQ |
|------|----------|-------|----------|
| 吞吐量 | 万级 | 百万级 | 十万级 |
| 延迟 | us级 | ms级 | ms级 |
| 适用 | 复杂路由 | 日志流 | 事务消息 |

**缓存策略**:
- Cache Aside: 读Cache miss查DB写Cache / 写DB后删Cache
- 缓存穿透: 布隆过滤器
- 缓存击穿: 热点Key加锁
- 缓存雪崩: 过期随机 + 多级缓存 + 熔断

**负载均衡**: DNS轮询 → L4(LVS) → L7(Nginx) → 客户端侧

## ❓ FAQ

| 问题 | 答案 |
|------|------|
| 单体转微服务时机 | 团队>30人/代码>20万行/部署周期>1周 |
| 同步vs异步 | 需实时结果用同步/可延迟用异步 |
| 分布式事务 | 强一致用TCC/大多数用最终一致/尽量避免跨服务事务 |
| 缓存一致性 | Cache Aside + 延迟双删 |
| 技术选型方法 | POC验证 → 矩阵打分 → 参考案例 → 团队能力 |
