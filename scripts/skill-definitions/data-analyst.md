## 📚 核心知识体系

### 数据分析完整工作流

**1. 问题定义(最关键!方向错了越努力越偏)**: 
- 业务方提出的需求通常是模糊的("帮我看看销售数据")→ 转化为可分析的具体问题
- 5W2H拆解: Who(分析对象)What(分析内容)When(时间范围)Where(范围)Why(目的)How(方法)How much(资源)
- 成功标准预设: 分析产出后业务方会做什么决策?(如果答案是不确定那可能需要重新定义问题)

**2. 数据采集**: 
- 内部数据: 数据库(MySQL/PostgreSQL)/业务系统导出/API接口/日志文件
- 外部数据: 公开数据集(政府统计局/行业报告)/第三方数据服务/网络爬虫(合规前提下)
- 数据质量检查: 完整性(有没有空值)/准确性(对不对)/一致性(同一事物不同地方记录是否一致)/时效性(是不是最新的)/唯一性(有没有重复)

**3. 数据清洗(占分析师60%+的时间!)**: 
- 缺失值处理: 删除/均值填充/中位数填充/前后值填充/建模预测填充
- 异常值检测: IQR方法/Z-score法/箱线图可视化/判断是真实异常还是错误需要领域知识
- 数据格式统一: 日期格式/数值单位/分类编码/文本标准化(大小写/空格)
- 去重: 精确去重/模糊去重(相似但不完全相同的记录)
- 数据类型转换: 字符串→日期/字符串→数字/宽表→长表(melt/unpivot)

**4. 探索性分析EDA**: 
- 描述性统计: 均值/中位数/众数/标准差/分位数/偏度峰度
- 分布可视化: 直方图/密度图/箱线图/小提琴图
- 相关性分析: Pearson/Spearman相关系数/相关热力图
- 分群分析: 按维度切片看分布差异(如按地区/产品线/用户类型)
- 时间趋势: 按日/周/月/季度看趋势/同比YoY环比MoM

**5. 建模与分析**(进阶): 
- 归因分析: 到底是什么导致了变化?(细分维度逐层下钻)
- 漏斗分析: 各步骤转化率/流失点识别
- 留存分析: Cohort同期群分析(按注册月份分组追踪留存)
- AB测试分析: 统计显著性检验(t检验/卡方检验)/效应量评估
- 预测基础: 时间序列/回归/简单分类(不需要成为数据科学家但要有基本概念)

**6. 可视化与沟通**: 
- 选择正确的图表(见下方图表选择指南)
- Dashboard设计原则(见下方)
- 数据讲故事(洞察→证据→建议→行动)
- 面向不同受众调整表达方式(高管要结论和建议/执行层要细节和方法)

### SQL核心查询模式

**必须掌握的SQL能力**: 

```sql
-- 基础查询
SELECT column1, COUNT(*) as cnt, AVG(amount) as avg_amt
FROM table_a a
LEFT JOIN table_b b ON a.id = b.a_id
WHERE date >= '2024-01-01'
GROUP BY column1
HAVING COUNT(*) > 10
ORDER BY cnt DESC
LIMIT 20;

-- 窗口函数(分析师必备!)
-- 同比
SELECT month, revenue,
       LAG(revenue, 12) OVER (ORDER BY month) as prev_year,
       revenue / LAG(revenue, 12) OVER (ORDER BY month) - 1 as yoy_growth
FROM monthly_revenue;

-- 用户留存(Cohort)
SELECT cohort_month,
       MONTH(add_months(cohort_month, n)) as activity_month,
       COUNT(DISTINCT user_id) as users,
       COUNT(DISTINCT user_id) / 
         FIRST_VALUE(COUNT(DISTINCT user_id)) OVER (PARTITION BY cohort_month) as retention_rate
FROM user_activity
GROUP BY cohort_month, n;

-- 排名
SELECT user_id, amount,
       RANK() OVER (ORDER BY amount DESC) as rank,
       NTILE(4) OVER (ORDER BY amount DESC) as quartile
FROM orders;
```

### 图表选择指南

| 想展示 | 推荐图表 | 避免 |
|--------|----------|------|
| 组成部分占比 | 饼图(≤5类)/堆叠条形图 | 饼图类别太多时 |
| 趋势变化 | 折线图 | 用柱状图表示连续时间 |
| 分类比较 | 条形图(横向)/柱状图(纵向) | 饼图用于比较 |
| 分布情况 | 直方图/箱线图 | 平均值掩盖分布 |
| 关系/相关性 | 散点图 | 不相关的变量硬找关系 |
| 地理数据 | 地图/热力地图 | 表格罗列地名 |
| 流程/转化 | 漏斗图/桑基图 | 复杂的文字描述 |
| 多维对比 | 雷达图(少量维度)/平行坐标 | 维度>8的雷达图 |

### Dashboard设计原则

**一屏原则**: 核心信息在一屏内可见不用滚动/层次分明(最重要的最大最显眼)
**5秒法则**: 5秒内理解当前状态是好是坏(使用红绿灯颜色编码/趋势箭头/目标对比)
**从宏观到微观**: 总览→筛选→详情(drill-down路径清晰)
**一致性**: 相同指标在不同页面用相同定义和计算方式/颜色含义一致(红色=坏绿色=好)
**更新频率合理**: 实时不一定好(导致过度反应)/根据决策频率设定刷新间隔
**移动适配**: 越来越多的高管在手机上看Dashboard/响应式设计或专门的移动视图

## 💼 实战案例

#### 电商业务数据体系从0到1搭建

**场景": 中型电商公司有交易数据但没有系统化的数据分析，每次要数据都找技术导Excel，效率低且容易出错，老板想要"随时能看到经营状况"

**方案**: 
1. 指标体系设计(OSM模型):
   - Objective目标: 提升公司整体盈利能力
   - Strategy策略1: 提高GMV → Measure: GMV/客单价/订单量/UV/转化率
   - Strategy策略2: 优化利润 → Measure: 毛利率/营销ROI/退货率/履约成本
   - Strategy策略3: 提升用户价值 → Measure: LTV/复购率/留存率/NPS
   - 北极星指标选定: GMV(短期)+ LTV(长期健康度)
2. 数据仓库分层:
   - ODS操作数据层: 原始数据原样接入(订单库/用户库/商品库/行为日志)
   - DWD明细数据层: 清洗后的明细表(每日订单明细/用户行为明细/商品明细)
   - DWS汇总数据层: 按主题汇总(日粒度的用户汇总/商品汇总/店铺汇总)
   - ADS应用数据层: 直接服务于报表和分析的宽表(管理层Dashboard直接查)
3. 核心报表建设(优先级排序):
   - P0 经营日报(CEO/COO): GMV/订单量/毛利率/UV/转化率 vs 目标vs 昨日/7日趋势
   - P1 商品分析(选品团队): 商品的销售额/库存/转化率/退货率/ABC分类
   - P1 用户分析(运营团队): 新增/活跃/留存/RFM分群/用户生命周期
   - P2 营销效果(市场团队): 渠道ROI/活动效果/获客成本/归因分析
   - P2 供应链(仓储物流): 库存周转/缺货率/发货及时率/退货处理时效
4. 数据治理:
   - 元数据管理(每个字段的业务含义/计算逻辑/负责人/更新频率)
   - 数据质量监控(每日自动化检查: 空值率/波动异常/数据量突降告警)
   - 权限控制(谁可以看什么/敏感数据脱敏)

**效果: 从"等3天拿数"到"实时看板" | 决策速度提升10倍 | 数据准确性问题减少90% | 团队从"问IT要数据"变为"自己看数据"

#### 用户流失分析与挽回方案

**场景": SaaS平台发现近3个月付费用户流失率从每月2%上升到5%，急需找出原因并制定挽回策略

**方案**: 数据驱动的流失诊断与干预
1. 流失定义与量化:
   - 定义流失: 连续30天未登录且已过续费期未续费的用户
   - 计算当前流失率: 确认5%的数据准确性(排除数据口径变化的影响)
   - 经济影响测算: 月流失5% × ARPU 200元 × 10000付费用户 = 月损失10万!
2. 流失画像分析(Who is churning?):
   - 按维度切分发现: 注册6-12月的用户流失率最高(12%)远高于新用户(3%)和老用户(2%) → "中年危机"现象
   - 行为特征: 流失前30天特征——登录频次下降40%/核心功能使用时长↓50%/客服工单数↑200%
   - 人口统计: 无显著差异(排除了特定人群的问题)
   - 归因假设: "用户在使用6-12个月后进入疲劳期/新鲜感消失而深度价值尚未充分感知"
3. 流失原因深挖(Why are they churning?):
   - 对已流失用户做抽样电话回访(N=50): Top 3原因 → ①"觉得没太大用了"(45%)②"找到了替代品"(28%)③"太贵了"(18%)
   - 结合行为数据交叉验证: "觉得没太有用"组确实深度功能使用率低 → 产品价值传递不足
   - 竞品分析: 发现竞品在"用户入职第3个月"推出了进阶引导而我们没有
4. 干预方案设计与A/B测试:
   - 实验组A(价值再激活): 在用户注册第6个月触发"进阶功能推荐"+ 专属客服主动触达
   - 实验组B(价格激励): 提供"续费9折"优惠券在到期前30天推送
   - 对照组: 无干预
   - 运行周期: 2个完整月
5. 结果:
   - A组流失率从5%降到2.8%(↓44%! 统计显著p<0.01)
   - B组流失率降到4.1%(↓18% 有效但不如A)
   - 结论: 价值再激活 > 价格激励(验证了"不是嫌贵是觉得不值")
   - 全量推广A方案 + 将其固化到产品流程中

**效果: 月流失率从5%稳定在2.5%左右 | 年度减少流失收入约90万 | 建立了完整的用户生命周期预警机制 | 产品团队据此开发了"用户成熟度引导"功能模块

## 🚀 进阶技巧

- 数据分析的核心不是技术而是商业理解: SQL和Python只是工具/真正有价值的是把数据和业务问题连接起来的能力/最好的分析师是最懂业务的人而不是代码写得最好的人/花80%的时间理解业务20%的时间写代码
- 先回答"So What?"再回答"Now What?": 数据告诉你发生了什么(So What = 这意味着什么?)→ Now What = 我们应该做什么?/只陈述事实不给出建议的分析师只能做一半的工作/永远以行动建议结尾
- 可视化的目的是降低认知负荷不是炫技: 最有效的图表往往是最简单的/去掉所有不必要的信息(网格线/3D效果/多余的颜色)/一个图表只讲一个故事/标注关键数据点和insight
- 数据质量是分析的生命线: Garbage In Garbage Out/在开始任何分析之前先评估数据质量/建立数据质量检查清单(完整性/准确性/一致性/时效性/唯一性)/发现数据问题时立即上报而不是默默继续
- 学会说不: 不是所有的分析需求都值得做/评估投入产出比(这个分析需要多少时间?能产生多少价值?)/帮助需求方重新定义问题(往往他们最初的问题不是真正需要解决的问题)/保护自己的时间和团队的资源

## ❓ 常见问题 FAQ

- Q: Excel够用吗还需要学SQL/Python? A: 取决于数据量和复杂度/<10万行Excel完全OK/>100万行或需要自动化/复杂分析则需要SQL(Python更进一步)/建议路线: Excel精通 → SQL掌握 → Python按需学习
- Q: 数据分析和数据科学家有什么区别? A: 分析师侧重描述性和诊断性分析(过去和现在发生了什么为什么)/数据科学家侧重预测性和指导性分析(未来会发生什么应该怎么做)/分析师更注重业务理解和沟通/科学家更注重算法和建模/两者边界越来越模糊
- Q: 怎么提升SQL能力? A: 多练! LeetCode SQL题库/HackerRank SQL/W3Schools实战练习/工作中遇到每个需求都用SQL解决而不是导出Excel/建立自己的SQL代码片段库(常用的查询模板)
- Q: Dashboard用什么工具好? A: Tableau(功能最强学习曲线陡)/Power BI(Microsoft生态集成好)/Metabase(开源免费适合内部工具)/FineReport(国内企业常用)/Google Data Studio(免费Google生态)/关键是先想清楚要展示什么再选工具
- Q: 如何证明数据分析的价值? A: 用数据证明数据的价值! 追踪你提供的分析带来的行动和结果(A/B测试是最好的方式)/记录节省的时间(以前人工X天现在自动Y分钟)/量化避免的错误(因为数据分析提前发现了问题)/定期汇报分析成果和价值案例
