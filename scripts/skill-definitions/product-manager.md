## 📚 核心框架

### 产品经理核心能力

```
硬技能:
├── 市场与用户研究: 竞品分析/用户访谈/问卷调查/数据分析
├── 需求分析与管理: 需求收集/优先级排序/变更控制
├── 产品设计: PRD撰写/原型制作/信息架构
├── 项目管理: 敏捷开发/路线图规划/跨部门协调
└── 数据分析: 指标定义/SQL基础/工具使用

软技能:
├── 沟通能力: 向上汇报/向下推动/横向协调
├── 同理心: 站在用户角度思考/理解不同角色诉求
├── 决策力: 信息不完整时做出合理判断/敢于取舍
└── 影响力: 无权领导/推动团队执行
```

### PRD产品需求文档

```
PRD标准结构:
├── 文档信息: 版本号/作者/日期/状态/变更记录
├── 背景与目标: 为什么做/成功标准是什么
├── 用户角色与场景: 谁会用/典型场景
├── 功能需求详细描述:
│   ├── 功能流程图: 主流程+异常流程+边界条件
│   ├── 页面原型: 线框图+交互说明
│   ├── 字段规则: 类型/长度/必填/默认值/校验规则
│   ├── 业务逻辑: 条件判断/计算公式/状态机
│   └── 权限设计: 哪些角色能看/操作什么
├── 非功能性需求:
│   ├── 性能要求: 响应时间/并发量
│   ├── 兼容性: iOS/Android/Web/浏览器版本
│   ├── 安全性: 数据加密/权限控制/日志审计
│   └── 可访问性
├── 数据埋点需求: 事件名/触发条件/携带参数
└── 验收标准: 测试用例要点/上线检查清单

User Story格式:
作为<角色>我希望<功能>以便<价值>
```

### 需求优先级决策

```
RICE评分法:
RICE = (R × I × C) / E

Reach覆盖范围: 影响多少用户?
├── 1 = 极少用户
├── 10 = 部分用户
├── 100 = 全部用户
└── 按实际用户数估算

Impact影响程度: 对单个用户价值多大?
├── 0.25 = 极小影响
├── 0.5 = 小影响
├── 1 = 中等影响
├── 2 = 大影响
└── 3 = 巨大影响

Confidence置信度: 对估计有多大把握?
├── 50% = 猜测
├── 80% = 有一定依据
└── 100% = 有数据支撑

Effort工作量: 需要多少人力时间?
├── 0.5 = 几小时
├── 1 = 几天
├── 2 = 几周
└── 按人月估算

分数越高优先级越高
```

### 产品生命周期

```
产品生命周期:
导入期 → 成长期 → 成熟期 → 衰退期

导入期策略:
├── 快速撇脂: 高价高促销
├── 缓慢撇脂: 高价低促销
├── 快速渗透: 低价高促销
└── 缓慢渗透: 低价低促销

成长期策略:
├── 改进产品质量
├── 进入新细分市场
├── 开拓新分销渠道
└── 适当降价

成熟期策略:
├── 市场改良: 寻找新用户/新用途
├── 产品改良: 提升质量/特性/样式
└── 营销组合改良: 价格/渠道/促销调整

衰退期策略:
├── 继续策略: 维持现状
├── 集中策略: 集中资源于有利市场
├── 收割策略: 减少投入/收割利润
└── 放弃策略: 停止生产/退出市场
```

### 数据指标体系

```
北极星指标North Star Metric:
├── 定义: 最能体现产品核心价值的单一指标
├── 特征: 反映用户价值/可衡量/可行动
└── 示例: Spotify听歌时长/Airbnb预订晚数

AARRR海盗指标:
├── Acquisition获取: 用户从哪里来?
│   └── 指标: CAC获客成本/渠道转化率
├── Activation激活: 首次体验核心价值了吗?
│   └── 指标: 激活率/Time to Value
├── Retention留存: 用户会回来吗?
│   └── 指标: D1/D7/D30留存率
├── Revenue变现: 如何赚钱?
│   └── 指标: ARPU/付费转化率/LTV
└── Referral推荐: 用户会推荐吗?
    └── 指标: K因子/推荐转化率

关键指标:
├── DAU/MAU: 日活/月活
├── DAU/MAU: 用户粘性(>20%为健康)
├── LTV/CAC: 用户终身价值/获客成本(>3为健康)
└── NPS净推荐值: 用户满意度(-100到100)
```

### MVP最小可行产品

```
MVP设计原则:
├── 最小: 只包含核心功能
├── 可行: 能够正常运行
└── 产品: 具备完整体验

MVP验证步骤:
├── 定义假设: 最关键的风险假设是什么?
├── 设计实验: 如何验证假设?
├── 构建MVP: 最小化实现
├── 收集数据: 用户行为数据
└── 学习迭代: 验证或调整假设

常见MVP类型:
├── 伪装门MVP: 只展示功能入口/实际未开发
├── 绿野仙踪MVP: 人工后台服务/模拟自动化
├── 单功能MVP: 只有一个核心功能
└── 精简版MVP: 功能完整但简化
```

### 敏捷开发流程

```
Scrum敏捷框架:
├── Product Backlog: 产品待办列表
├── Sprint: 2-4周迭代周期
├── Sprint Planning: 迭代规划会
├── Daily Standup: 每日站会(15分钟)
├── Sprint Review: 迭代评审会
└── Retrospective: 迭代回顾会

产品路线图Roadmap:
├── 战略层: 1-3年愿景
├── 规划层: 季度目标
├── 执行层: 迭代计划
└── 调整频率: 每季度调整

需求池管理:
├── 需求收集: 多渠道收集
├── 需求分析: 价值/成本评估
├── 优先级排序: RICE/Kano模型
└── 迭代排期: 根据团队容量
```
